# UserScenarios — 사용자 시나리오

> IA.md / Personas.md와 독립 관리. 화면 번호는 IA.md의 SC-xx를 참조.

## US-01 태스크 제출과 협업 관찰 (핵심 경로)

1. 사용자가 로컬 서버를 실행하고 브라우저에서 대시보드(SC-01)를 연다.
2. 태스크를 입력한다. 예: "국내 중소기업용 재고관리 SaaS 아이디어를 평가하고 요약 보고서를 만들어줘."
3. 팀 설정(기본 팀)을 선택하고 제출한다.
4. 세션 상세(SC-03)로 이동하면 에이전트들이 기동되고, 서로 메시지를 주고받는 과정이
   타임라인에 실시간으로 흐른다 (조사자가 정리 → 분석가가 반박/보완 → 작성자가 종합).
5. 한 에이전트가 `submit_result`로 초안을 제출하면 다른 에이전트들이 승인/반대를
   투표하고(화백 합의, 기본 만장일치 — D-011), 승인되면 세션이 완료되고 결과 카드가 표시된다.
6. 사용자는 결과를 복사해 사용한다.

**성공 기준**: 제출부터 결과까지 개입 없이 완료. 과정이 실시간으로 보임.

## US-02 진행 중 세션 취소

1. US-01 진행 중, 사용자가 에이전트들이 잘못된 방향으로 논의 중임을 발견한다.
2. 세션 상세(SC-03)의 취소 버튼을 누른다.
3. 세션이 `cancelled` 상태로 종료되고, 진행 중이던 API 호출은 중단되며 그 시점까지의
   토큰 사용량이 표시된다.

**성공 기준**: 취소 후 추가 API 호출·비용 발생 없음.

## US-03 예산 초과 자동 종료 (실패 경로)

1. 에이전트들이 결론 없이 긴 논의를 이어간다.
2. 작업량이 설정 임계값에 도달하면 시스템이 종합과 제안을 우선하고 일반 토론을
   닫는다. 일반 채팅 수가 먼저 상한에 도달해도 실패시키지 않고 제안 단계로
   전환하며, 제안·투표 메시지는 채팅 상한에서 제외한다. 캐시 읽기는 작업 예산에서
   제외하지만 전체 처리 상한에는 포함한다.
3. 작업 예산 또는 전체 처리 상한을 넘으면 시스템이 세션을 강제 종료한다.
4. 세션 상태가 `failed (budget)`로 표시되고, 그때까지의 메시지 이력과 작업/캐시/
   전체 처리 사용량이 남는다.

**성공 기준**: 무한 대화로 비용이 새지 않음. 실패 원인이 명확히 표시됨.

## US-03a 안건에 맞는 예산 프로필 선택

1. 사용자는 가역적인 짧은 선택이면 Quick, 일반 기술 의사결정이면 Standard(default),
   고위험·비가역적 결정이면 Deep을 선택한다.
2. 홈의 팀 설명과 팀 화면의 예산·에이전트 구성을 확인하고 태스크를 제출한다.
3. 선택한 프로필의 작업/전체 처리 상한과 단계별 수렴 규칙이 세션에 적용된다.

**성공 기준**: 사용자가 YAML을 편집하지 않고도 품질과 비용의 균형을 선택할 수 있음.

## US-04 팀 구성 변경

1. 사용자가 `configs/` 아래에 새 팀 YAML을 작성한다 (예: 비평가 역할 추가).
2. 서버가 요청마다 `configs/`를 다시 읽으므로 재시작 없이 팀 목록(SC-04)에 새 팀이 나타난다.
3. SC-01에서 새 팀을 선택해 태스크를 제출한다.

**성공 기준**: 설정 파일만으로 팀 구성 변경 가능. 잘못된 YAML은 명확한 검증 오류로 안내.

## US-05 세션 히스토리 확인

1. 사용자가 며칠 전 실행했던 세션의 결과를 다시 보고 싶다.
2. 세션 목록(SC-02)에서 해당 세션을 찾아 클릭한다.
3. 메시지 이력 전체와 최종 결과가 그대로 복원되어 표시된다.

**성공 기준**: SQLite 영속화로 서버 재시작 후에도 세션 이력이 보존됨.

## US-06 초안 반려와 재제출 (합의 경로)

1. US-01 진행 중 작성자가 `submit_result`로 초안을 제출한다 — 세션이 voting 상태가 되고
   투표 현황이 표시된다.
2. 분석가가 근거 부족을 사유로 reject 투표를 한다 — 세션이 running으로 복귀하고,
   반려 사유가 타임라인에 표시되며 작성자에게 전달된다.
3. 작성자가 보완해 재제출하고, 전원 승인으로 세션이 완료된다.

**성공 기준**: 반려 사유가 타임라인에 보이고, 재제출 과정에도 메시지/토큰 예산이 계속 강제됨.
