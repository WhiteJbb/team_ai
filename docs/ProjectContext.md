# ProjectContext — 프로젝트 배경과 목표

## 프로젝트 개요

여러 Claude 에이전트가 **서로 직접 메시지를 주고받으며(자율 협업/메시지 패싱)** 사용자가
제출한 범용 태스크를 분담·협업 처리하는 **멀티 에이전트 오케스트레이션 시스템**을 만든다.

- Claude Code의 내장 기능(Agent Teams, 서브에이전트)을 사용하는 것이 아니라,
  **Claude API(Messages API + tool use) 위에 오케스트레이션 계층을 직접 구현**한다.
- 목적은 학습용 데모가 아닌 **실제 사용할 도구**다.
- 사용 형태: **로컬 서버 + 웹 대시보드**. 사용자는 브라우저에서 태스크를 제출하고,
  에이전트들의 메시지 교환·작업 진행을 실시간으로 관찰하고, 결과를 수령한다.

## 배경 (2026-07-14)

- 사용자가 "멀티 에이전트 팀을 만들고 싶다"고 요청. 처음에는 Claude Code 내장 기능
  (Agent Teams 실험 기능, `.claude/agents/` 서브에이전트) 안내가 이루어졌으나,
  사용자의 의도는 **자체 멀티 에이전트 오케스트레이션 시스템을 직접 만드는 것**으로 확인됨.
- 이후 6가지 방향 질문을 통해 아래와 같이 결정됨 (근거는 `DecisionLog.md` 참조):

| 항목 | 결정 |
|---|---|
| 목적 | 실제 사용할 도구 |
| 기반 기술 | Claude API 직접 구현 (Agent SDK / Managed Agents 아님) |
| 언어 | Python |
| 오케스트레이션 패턴 | 자율 협업 (에이전트 간 메시지 패싱) |
| 처리 대상 | 범용 태스크 (특정 도메인 없음) |
| 인터페이스 | 로컬 서버 + 웹 대시보드 |

## 핵심 설계 과제

자율 협업(메시지 패싱) 패턴은 중앙 조율자가 없거나 약하기 때문에, 다음이 핵심 난제다:

1. **수렴/종료 제어** — 에이전트들이 무한 대화에 빠지지 않도록 종료 조건
   (결과 제출 도구, 최대 메시지 수, 토큰 예산, 유휴 감지)을 시스템 차원에서 강제해야 한다.
2. **비용 통제** — 에이전트 수 × 대화 길이만큼 API 비용이 곱해진다. 토큰 예산과
   프롬프트 캐싱 설계가 필수다.
3. **관측성** — 메시지 흐름이 비선형이므로 대시보드에서 세션/에이전트/메시지를
   시각적으로 추적할 수 있어야 한다.

## 제약 / 환경

- Windows 11 + PowerShell 5.1, 콘솔 cp949 → **사용자 노출 문자열(로그, CLI 출력)은 영어 ASCII**,
  주석·독스트링은 한국어 (CLAUDE.md 규칙).
- Python 가상환경(.venv) + unittest.
- API 키는 로그·에러 응답·대시보드 어디에도 노출 금지(마스킹).
- 요청 경로 코드는 블로킹 I/O 금지.
