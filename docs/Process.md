# Process — 작업 방식과 절차

> 이 프로젝트에서 일하는 방식. 상위 규칙은 루트 `CLAUDE.md`가 원본이며, 이 문서는 실행 절차로 구체화한 것.

## 작업 사이클

1. **계획** — 구현 전 `docs/Plan.md`를 갱신하고 그 계획대로 구현한다. 계획 없는 구현 금지.
2. **질문** — 기술 스택·정책·범위 등 사용자 결정이 필요한 사항은 임의 진행하지 않고 먼저 묻는다.
   결정되면 `docs/DecisionLog.md`에 기록(무엇을/왜/대안).
3. **브랜치 작업** — 큰 기능/변경은 `feat/<기능명>` 등 브랜치에서 작업 → PR → **squash merge**
   → 브랜치 삭제. 문서만 수정하는 변경은 main 직접 커밋 허용.
4. **기록** — 진행한 작업·오류·수정 내역을 `docs/WorkLog.md`에 남긴다.
5. **검토** — 마일스톤 완료 시 3렌즈 병렬 검토(코드 정밀/신규 사용자 워크스루/문서-코드 일치성)
   후 발견 사항 일괄 수정. 머지 전 `docs/ReviewChecklist.md` 점검.

## 구현 원칙 (요약)

- **계약 우선**: 타입·프로토콜·설정 스키마를 먼저 확정·커밋하고, 모듈은 그 위에서 병렬 구현.
- **직접 + 위임 하이브리드**: 아키텍처(계약, 조립, 통합·리뷰)는 직접, 명세가 명확한
  독립 모듈은 서브에이전트 병렬 위임(파일 소유권 분리, 검증 명령 명시).
- **추측 금지**: 버그는 최소 재현으로 원인 확정 후 수정.
- **실패 경로가 제품**: 새 기능에는 실패 시나리오 테스트를 함께 만든다.
- **테스트 밀폐**: 실키·실네트워크 의존 금지(Fake LLM 클라이언트), 실스택 E2E는 별도 스모크로.

## 환경 절차 (Windows / PowerShell 5.1)

- 테스트: `.venv\Scripts\python.exe -m unittest discover -s tests`
- git 인자에 큰따옴표 금지 (PS 5.1 네이티브 인자 전달 문제).
- 사용자 노출 문자열은 영어 ASCII (콘솔 cp949), 주석·독스트링은 한국어.
- 커밋/PR에 AI 흔적(Co-Authored-By, 생성 도구 서명) 금지.
