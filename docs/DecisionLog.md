## D-011 (2026-07-14) 결과 확정: 화백 합의 모드 (투표 승인)

- **결정**: `submit_result`는 즉시 종료가 아니라 **합의 절차**를 연다. 제출된 초안이
  전원에게 브로드캐스트되고, 다른 에이전트들이 `vote_result`(approve/reject + 사유)로
  투표한다. 정족수는 팀 설정 `termination.approval`로 지정 — 기본 **unanimous**
  (제출자 제외 전원 승인), 옵션 `majority`(과반) / `first`(선착순, 투표 생략).
- **대안**: (a) 선착순 — 단순하지만 품질 검증 없음, (b) 선착순 + 합의는 추후 옵션.
- **근거**: 사용자 선택. 프로젝트 이름(화백 = 만장일치 회의체)의 정체성에 부합하고,
  결과 품질 검증(상호 비판)이 시스템에 내장됨. 순회 비용 증가와 교착 위험은
  기권 처리·no_quorum 실패·예산 상한으로 방어 (Plan "코어 의미론" §5).
- **결정자**: 사용자.
- **개정**: D-016에서 정족수 의미론 개정 — "기권 제외 전원 승인" 규칙은 엄밀한
  만장일치가 아니어서 `participating_unanimous`로 분리되고, 기본 `unanimous`는
  생존 심의자 전원 approve로 엄밀화됨.

## D-015 (2026-07-14) Hermes 미도입 — 화백 코어 직접 구현 유지

- **결정**: Hermes(외부 에이전트 런타임/워커 프레임워크) 기반으로 전환하지 않고
  기존 화백 코어(세션 상태 기계, asyncio 메시지 버스, 인박스 배치 소비, 종료 정책,
  제안·투표·반려·재합의, 도메인 이벤트, 결정 기록)를 직접 소유·구현한다 (D-001 재확인).
- **근거**: Hermes는 장기 실행 작업·작업 큐·프로필별 워커에 강점이 있으나, 화백의
  핵심인 실시간 세션 협업·메시지 배치 소비·합의 의미론과는 모델이 다름. 지금 전환하면
  의존성만 늘고 화백 전용 로직은 대부분 재구현 필요.
- **후속 조치**: 현재 마일스톤에서 도입하지 않음. 후순위 실험 대상으로만 기록 —
  향후 웹 조사/코드 수정/장기 실행 작업자가 필요할 때 외부 Agent Runtime 또는
  Worker Adapter로 검토. Hermes SDK나 전용 타입을 코어 계약에 노출하지 않는다.
  에이전트 실행부는 세션·합의 로직과 결합도를 낮게 유지하되(추후 AgentRuntime
  Protocol 추출 가능하게), 현재는 별도 Runtime 추상화를 만들지 않는다.
- **결정자**: 사용자.

## D-016 (2026-07-14) 합의 의미론 개정: 정족수 4종 + 제안 버전 + voting 잠금 (D-011 개정)

- **결정**:
  1. 정족수 정책을 4종으로 구분한다 — `unanimous`(기본): 제출자 제외 **생존 심의자
     전원** approve. `majority`: 생존 심의자 **전체의 과반** approve(유효 투표 과반
     아님). `participating_unanimous`: 유효 투표자 전원 approve(구 D-011 규칙이 이 이름
     으로 이동). `first`: 투표 없이 즉시 확정.
  2. **미투표는 암묵적 승인이 아니다** — unanimous에서 투표 시간 만료 시 미투표자가
     있으면 승인하지 않고 no_quorum 경로로 처리.
  3. **제안 버전**: `ResultProposal`(id/session_id/proposer/version/content/created_at)
     도입. 모든 투표는 proposal_id 필수, 이전 제안에 대한 늦은 투표는 무시(엔진 규칙).
     반려 후 재제출은 version을 올린 새 제안.
  4. **voting 잠금**: submit_result는 running에서만, voting 중에는 현재 제안에 대한
     vote_result만 허용, voting 중 추가 submit_result는 도메인 오류로 거부(엔진이
     강제, 상태 기계의 VOTING→VOTING 금지가 계약 차원 방어).
  5. ~~심의 중 사망자는 심의 대상에서 제외해 정족수 재계산(`with_voter_removed`)~~
     — D-018에서 번복(스냅샷 불변).
- **근거**: 사용자 설계 검토 — D-011의 "기권 제외 전원 승인"은 엄밀한 만장일치가
  아님(침묵이 승인으로 둔갑). 화백의 정체성인 만장일치를 기본값에서 엄밀하게 지키고,
  완화가 필요한 팀은 설정으로 명시적으로 선택하게 한다. 제안 버전은 반려 후 재제출과
  늦은 투표의 안전한 처리에 필수.
- **결정자**: 사용자.

## D-018 (2026-07-14) 투표 대상자: voting 시작 시점 스냅샷 불변 (D-016 §5 번복)

- **상태**: 채택.
- **맥락**: 심의 도중 에이전트가 사망하거나 LLM 호출에 묶여 투표하지 못하는 경우,
  정족수의 분모를 어떻게 다룰지가 모호했다. D-016은 사망자를 심의 대상에서 제외해
  재계산(`with_voter_removed`)하기로 했었다.
- **결정**: voting 시작 시점의 **생존 에이전트(제출자 제외)를 투표 대상자로 확정
  (스냅샷)**하고, 심의 도중 사망해도 집합을 변경하지 않는다. 투표 불가(사망·오류)는
  voting_timeout 만료 시 기권으로 처리되어 unanimous에서는 no_quorum으로 이어진다.
  투표 대상자가 0명이면 first 모드가 아닌 한 no_quorum이며, 팀 검증 단계에서
  "first가 아닌 모드 + 2인 미만 팀"을 사전 거부한다.
- **대안**: (a) 사망 시 재계산(구 D-016) — 분모가 흔들려 "사망 타이밍에 따라 승인
  여부가 바뀌는" 비결정성 발생, (b) 사망 시 즉시 no_quorum — 재시도 여지 없이 과격.
- **선택 이유**: 판정의 결정성(같은 투표 = 같은 결과)과 감사 가능성. 화백의 만장일치
  정신에서 "심의자가 빠졌으니 남은 사람만으로 합의"는 정족수 완화의 뒷문이 된다.
- **결과와 트레이드오프**: `with_voter_removed` 계약 삭제. 사망자가 있으면
  unanimous 제안은 사실상 실패하고 세션이 no_quorum으로 끝난다 — 엄격하지만
  명시적. 완화가 필요한 팀은 participating_unanimous를 선택.

## D-019 (2026-07-14) 타이머 분리: idle_timeout / voting_timeout + approval 구조형 설정

- **상태**: 채택.
- **맥락**: idle_timeout 하나로 running 유휴 감지와 voting 투표 대기를 겸용하면,
  투표 대기 중 idle 판정이 세션을 failed(idle)로 종료하는 레이스가 생긴다.
- **결정**: `termination.idle_timeout`은 running 전용(전원 인박스 비고 호출 없음),
  `approval.voting_timeout`은 voting 전용(미완료 투표 대기)으로 분리한다. voting
  중에는 idle 감시가 세션을 종료하지 않는다. 두 타이머 모두 세션의 단일 감시
  주체가 관리한다. 팀 설정의 approval은 문자열 하나에서 구조형으로 확장:
  `approval: {mode, timeout_seconds, minimum_votes}` — 기존 문자열은 mode 축약형으로
  하위호환 유지(마이그레이션: config.py 독스트링 문서화).
- **대안**: 단일 타이머 유지(구 Plan) — 레이스 원인. voting 무제한 대기 — 교착.
- **선택 이유**: 상태별 의미가 다른 대기를 한 값으로 묶으면 튜닝 불가(빠른 idle 감지
  vs 충분한 심의 시간은 상충).
- **결과와 트레이드오프**: 계약 `ApprovalConfig` 신설, TerminationPolicy.approval
  타입 변경(하위호환은 로더가 흡수). 설정 항목이 하나 늘어남.

## D-021 (2026-07-14) 세션 종료 직렬화·우선순위 + interrupted

- **상태**: 채택 (우선순위의 실효성은 M2 경쟁 조건 테스트에서 재검증).
- **맥락**: 마지막 투표와 예산 초과, 취소와 승인 등 종료 조건이 동시에 발생할 수
  있다. 종료가 두 번 확정되면 결과/사유가 비결정적이 된다.
- **결정**: 종료 전환은 세션 단위 lock(단일 coordinator)이 직렬화하고, 최초로
  확정된 유효 종료 사유만 저장한다. 경합 시 우선순위:
  **cancelled → completed → budget/messages → agent_error → no_quorum → idle**.
  종료 후 도착한 이벤트는 상태를 변경하지 못하며 감사용 rejected event로 기록할
  수 있다. 서버 재시작 시 이전 running/voting 세션은 **failed(interrupted)**로
  처리한다 — `FailReason.INTERRUPTED` 신설 (agent_error에 뭉뚱그리지 않음,
  오류 귀책 정확성 원칙).
- **대안**: 도착 순 선착(비결정적), interrupted 없이 agent_error 재사용(귀책 왜곡).
- **선택 이유**: 사용자 의사(취소)와 이미 성립한 합의(completed)가 시스템 사정보다
  우선해야 관측 결과가 직관과 일치한다.
- **결과와 트레이드오프**: 우선순위 표가 계약 문서에 고정됨 — 실제 경쟁 조건에서
  이 순서가 필요한지 M2에서 테스트로 재검증하고, 불필요하면 단순화한다.

     프로필로 제공해 작업 성격에 따라 명시적으로 선택한다.
- **대안**: 캐시를 포함한 기존 100k 단일 상한 유지(캐시 재사용을 벌점화), 상한만

## D-033 (2026-07-14) 토론 한도와 결정 한도 분리

- **상태**: 채택 (사용자 결정).
- **맥락**: M5.1 병합 후 첫 Standard 실안건에서 작업 43,324에 proposal 단계로
  전환했지만, 진행 중 synthesis 호출과 한 응답의 다중 `send_message`가 일반 채팅
  30개를 먼저 채웠다. 상대등은 작업 56,688에 정상 제안 v1을 제출했으나
  `result_proposal`이 31번째 메시지로 계산되어 즉시 `failed(messages)`가 됐고
  투표는 시작하지 못했다. 심의자 한 명은 늦은 synthesis 호출로 12번째 턴까지
  소비해 결정 단계 직전에 `max_turns`도 소진했다.
- **결정**:
  1. `max_messages`는 일반 `chat`의 상한으로 정의한다. 제안·투표는 결정 제어
     메시지이므로 제외하고, 상한 도달은 실패가 아니라 proposal 단계 강제 전환으로
     사용한다. 이후 일반 채팅은 거부한다.
  2. synthesis/proposal 임계값은 사후 관측점이 아니라 다음 호출의 입장 게이트다.
     현재 작업량·진행 중 예약·다음 예약을 합쳐 임계값을 넘길 호출은 받지 않고,
     진행 중 호출 정산 후 다음 단계로 전환한다.
  3. 한 LLM 응답에서는 `send_message`를 최대 한 번만 실행한다. 여러 수신자는 기존
     `recipients` 배열로 묶으며 중복 호출은 구조화된 도구 오류로 돌려준다.
  4. `max_turns`는 열린 토론의 상한으로 유지하되, 한도 도달 에이전트를 즉시 결정
     자격에서 제거하지 않는다. proposal/voting/revision은 단계·제안별 최대 2회라는
     별도 상한 안에서만 추가 호출을 허용한다. 제출자가 토론 턴 한도에 닿으면
     기다리지 않고 proposal로 전환한다.
- **대안**: `max_messages`·`max_turns`·작업 예산을 단순 상향(수렴 실패 은폐),
  제안 메시지만 한도에서 예외 처리(다중 채팅과 늦은 단계 전환 미해결), 진행 중
  LLM 호출 강제 취소(usage 유실·예약 정리 경합 위험).
- **결과와 트레이드오프**: 토론 폭주 상한은 유지하면서 이미 생성한 제안과 투표가
  토론 한도에 막히지 않는다. 결정 단계의 추가 호출은 기존 2회 제한으로 유한하다.
  진행 중 호출은 취소하지 않아 일부 토큰 초과 가능성은 남지만 예약 게이트로 경계
  근처 동시 호출을 줄인다.
- **결정자**: 사용자 (첫 M5.1 실안건 분석 후 승인).

  1. voting 호출 입장 시 활성 proposal id를 에이전트 호출 문맥에 캡처한다.
     `vote_result`가 id를 생략해도 현재 활성이 아니라 이 캡처 id로 등록한다.
  2. 승인·반려·정족수 없음으로 제안이 해소되면 해당 proposal id를 검토 중인
     나머지 LLM 호출만 취소한다. outer 에이전트 루프는 유지하고 취소 경로에서
     호출 예약을 반환해 후속 제안 알림과 투표를 처리하게 한다.
  3. provider가 취소를 늦게 반영해 낡은 응답을 돌려도 캡처 id가 superseded이면
     투표를 무시하고 활성 제안 정보를 반환한다.
- **대안**: 작업 예산 또는 예약 크기를 10토큰 조정(동일 경합 재발), 모든 진행
  호출 취소(D-033의 usage·토론 손실 위험), id 생략 금지(모델의 id 오기입 회귀).
- **결과와 트레이드오프**: 반려 직후 낡은 네트워크 요청과 예약을 회수해 v2가
  남은 결정 예산을 쓸 수 있고, 늦은 응답의 세대 오염을 막는다. LLM 호출을 자식
  태스크로 감싸므로 fake처럼 즉시 반환하는 구현에서도 한 번의 스케줄 지점이
  추가되며, 상태 전환 경계의 도구 오류는 도구 미제공 또는 상태 거부 둘 다 가능하다.
- **결정자**: 사용자 (`f26e2d65-000000` 분석 후 수정 승인).

## D-035 (2026-07-15) voting 호출의 제안 본문 스냅샷 보장

- **상태**: 채택 (사용자 결정).
- **맥락**: D-034 병합 후 실안건 `56c92390-000000`은 완료됐지만 critic이 v1을
  "활성 제안 본문이 제공되지 않았다"는 이유로 반려했다. v1 본문 3,826자는 버스와
  DB에 정상 존재했고 research는 같은 제안을 승인했다. AgentLoop가 tool-use 연속
  호출 안에 있으면 outer wait/drain으로 돌아오기 전에 예산 단계만 voting으로 바뀔
  수 있어, 투표 도구와 단계 지시는 받지만 제안 본문은 이력에 없는 호출이 가능했다.
  반대로 discussion에서 이미 시작된 요청이 제안 생성 후 반환한 `vote_result`가
  현재 voting에 우연히 적용되는 경계도 기존 테스트에서 확인했다.
- **결정**:
  1. voting 호출 입장 시 proposal id만이 아니라 불변 `ResultProposal` 전체를 호출
     문맥에 캡처한다. 요청 이력에 해당 proposal marker가 없으면 동일한 제안 본문과
     행동 지시를 직접 주입하고, 이미 inbox로 받은 경우에는 중복하지 않는다.
  2. 매 LLM 호출 입장 승인 직후 inbox를 drain해 연속 tool-use 중 도착한 제안·투표·
     제어 알림을 다음 요청에 반영한다.
  3. voting 상태에서 `vote_result`를 실행하더라도 그 LLM 호출의 시작 단계가
     discussion/synthesis였다면 거부한다. 도구 오류를 받은 루프는 정상 voting
     호출로 다시 입장해 본문을 검토한 뒤 투표한다.
- **대안**: voting 지시에 본문을 항상 중복 첨부(토큰 낭비), outer loop에서만 inbox
  소비 유지(실패 재발), 상태가 voting이면 오래된 호출의 투표도 허용(본문 없는
  오판 및 세대 경계 약화).
- **결과와 트레이드오프**: 모든 정상 voting 호출은 검토할 정확한 제안 본문을
  보장받고, 단계 전환에 걸친 오래된 응답은 결정에 반영되지 않는다. 후속 호출이
  inbox를 더 일찍 소비하므로 실시간 협업 문맥은 좋아지지만 기존 fake 테스트 중
  오래된 discussion 투표 허용에 의존한 스크립트는 명시적 단계 대기로 수정했다.
- **결정자**: 사용자 (`56c92390-000000` 분석 후 수정 승인).

## D-036 (2026-07-15) 위험도 기반 main 직접 커밋 허용

- **상태**: 채택 (사용자 결정).
- **맥락**: 기존 규칙은 문서 외 모든 코드 변경에 브랜치와 PR을 요구해, 오탈자·
  메시지·테스트 fixture·명백한 국소 수정에도 게시 절차 비용이 동일했다. 반면 최근
  voting 수정처럼 변경량이 작아 보여도 동시성·상태 전이 의미론을 바꾸는 작업은
  독립 검토와 롤백 경계가 필요했다.
- **결정**: 파일 수나 diff 크기만이 아니라 실패 영향과 롤백 난이도로 워크플로우를
  선택한다. 문서·표시·제품 의미론 없는 테스트 보정과 고위험 영역을 건드리지 않는
  1~2개 파일의 명백한 국소 버그는 관련 테스트 후 main 직접 커밋/푸시를 허용한다.
  새 기능, 여러 모듈 동작, 계약·DB·설정·의존성, 비동기/동시성·자원 관리,
  상태 전이·합의·예산, 인증·보안·마이그레이션은 계속 브랜치와 PR을 강제한다.
- **안전 규칙**: 애매하면 PR을 사용하고, main 직접 변경은 한 커밋으로 되돌릴 수
  있는 작은 범위와 명시적 diff·테스트 확인을 전제로 한다.
- **결정자**: 사용자.

## D-037 (2026-07-15) 메시지 뒤 대기와 voting 문맥 압축

- **상태**: 채택 (사용자 결정).
- **맥락**: 실안건 `880ec321-000000`은 제안과 두 심의 응답까지 생성했지만 작업
  62,120/60,000으로 종료됐다. critic은 동일한 릴리스 게이트 의견을 11회 반복해
  작업 34,683을 썼고, proposal은 작업 46,327에서 voting에 진입했다. 두 투표
  호출은 각각 6k를 예약해 입장했지만 실제 합계가 15,793으로 예약보다 커 최종
  정산에서 예산을 넘었다. 투표 출력은 짧았고 누적 토론 입력 재전송이 주원인이었다.
- **결정**:
  1. 오류 없이 도구를 실행하고 추가 결정 행동이 요구되지 않으면 AgentLoop의
     연속 tool-use 호출을 끝내고 outer inbox 대기로 복귀한다. 특히 성공한
     `send_message` 자체는 다음 추론의 입력 신호가 아니며, 새 메시지가 와야 깨어난다.
  2. 도구 오류 또는 아직 기록되지 않은 투표·제출에는 기존 단계별 한도 안에서
     교정 호출을 허용한다.
  3. voting 요청은 전체 토론 대신 원래 작업, 호출에 캡처한 활성 제안, 같은 제안에
     대한 직전 도구 피드백만 전달한다. 제안이 바뀌면 이전 피드백은 폐기한다.
- **대안**: Standard 작업 예산 또는 호출 예약을 단순 상향(반복과 누적 문맥 비용을
  숨김), 토론을 모델 요약으로 압축(추가 호출·왜곡 위험), 투표 피드백까지 제거
  (잘못된 id나 단계 경합의 교정 능력 저하).
- **결과와 트레이드오프**: 에이전트는 메시지를 보낸 뒤 동료 입력을 기다리므로
  불필요한 자기 연속 호출이 줄고, 투표 입력 비용은 토론 길이와 분리된다. 동료가
  답하지 않으면 에이전트도 진행하지 않지만 기존 idle 종료가 이를 유한하게 막으며,
  결과를 독자적으로 확정할 수 있는 에이전트는 처음부터 `submit_result`를 사용할 수 있다.
