# Working Rules (프로젝트 작업 규칙)

이 프로젝트에서 작업할 때는 아래 규칙을 반드시 따른다.

1. **배경/중요 대화 기록**: 프로젝트 배경과 중요한 논의 내용은
   `docs/ProjectContext.md`에 기록한다. 새로운 배경 정보나 방향 전환이
   생기면 해당 문서를 갱신한다.
2. **구현 전 Plan.md 작성**: 실제 코드 구현을 시작하기 전에 반드시
   `docs/Plan.md`를 작성(또는 갱신)하고, 그 계획에 따라 구현한다.
   계획 없이 바로 구현을 시작하지 않는다.
3. **기획 문서 독립 관리**: `docs/IA.md`(정보 구조),
   `docs/UserScenarios.md`(사용자 시나리오), `docs/Personas.md`(페르소나)는
   서로 합치지 않고 독립된 문서로 관리한다.
4. **사용자 결정 필요 시 질문**: 기술 스택, 정책, 범위 등 사용자의
   결정이 필요한 사항은 임의로 진행하지 않고 먼저 질문한다.
5. **의사결정 기록**: 기술 스택 선택, 정책 결정 등 주요 의사결정은
   `docs/DecisionLog.md`에 남긴다(무엇을, 왜 그렇게 결정했는지 포함).
6. **작업/오류 기록**: 진행한 작업, 발생한 오류와 그 수정 내역은
   `docs/WorkLog.md`에 기록한다.
7. **브랜치/PR 워크플로우**: 큰 기능이나 변경은 `main`에 직접 커밋하지
   않는다. 반드시 브랜치를 파서 작업하고 PR을 거쳐 병합한다.
   - "큰 기능/변경" 기준: 새 기능 구현, 여러 파일에 걸친 코드 변경,
     구조/스키마/설정 형식 변경, 마일스톤 단위 작업
   - 브랜치 네이밍: `feat/<기능명>`, `fix/<대상>`, `refactor/<대상>`, `docs/<대상>`
   - 작업 완료 후 PR 생성(`gh pr create`) — 본문에 변경 요약과 테스트/검증
     결과를 포함한다
   - 병합은 **squash merge**(`gh pr merge --squash`)로 한다 — main 히스토리는
     PR 단위 1커밋으로 유지
   - 병합 후 작업 브랜치는 삭제한다
   - 예외: **문서만 수정하는 변경**(코드 미포함)은 main에 직접 커밋/푸시 허용
8. **Git 작업에 AI 흔적 금지**: 커밋 메시지, PR 제목/본문, 브랜치명 등
   어떤 git 산출물에도 AI 사용 흔적을 남기지 않는다.
   - `Co-Authored-By: Claude ...` 트레일러 금지
   - `🤖 Generated with Claude Code` 같은 생성 도구 서명/링크 금지

# Development Guide (개발 지침)

## 구현 방식

- **계약 우선**: 타입·프로토콜·설정 스키마를 먼저 확정해 커밋하고, 모듈들은
  그 계약 위에서 병렬로 구현한다. 계약 없이 모듈부터 만들면 통합에서 깨진다.
- **직접 + 위임 하이브리드**: 아키텍처가 걸린 부분(계약, 선택/failover 파이프라인,
  서버 조립, 통합·리뷰)은 직접 작성하고, 명세가 명확한 독립 모듈은 서브에이전트에
  병렬 위임한다. 위임 시 반드시: 파일 소유권을 겹치지 않게 분할, 프롬프트에
  읽을 파일·계약·검증 명령·최종 보고 형식 명시, 같은 파일을 동시 편집하지 않기.
- **마일스톤 완료 시 렌즈별 병렬 검토**: 코드 정밀(버그) / 신규 사용자 워크스루(UX) /
  문서-코드 일치성 — 세 관점을 독립 실행하고 발견을 일괄 수정한다.
  한 관점의 리뷰는 다른 관점의 결함을 보지 못한다.

## 검증 원칙

- **추측 금지, 실증 우선**: 버그·이상 동작은 최소 재현이나 직접 호출로 원인을
  확정한 뒤 고친다. "아마 이것 때문"으로 수정하지 않는다.
- **실패 경로가 제품이다**: 새 기능에는 실패 시나리오(타임아웃/절단/한도 초과/취소)
  테스트를 함께 만든다. 대역(Fake)이 통과해도 실스택 E2E(시뮬레이터)를 신뢰 기준으로
  삼는다 — 대역은 실스택의 미묘한 동작(usage 청크 합성, 헤더 위치 등)을 가린다.
- **오류 귀책을 정확히**: 실패를 기록할 때 누구 잘못인지 구분한다 — 클라이언트의
  잘못된 요청이나 게이트웨이 자체 혼잡을 모델 health에 묻히면 자가 치유가 오작동한다.
- **머지 전 필수**: 전체 테스트 + 통합 스모크 통과. 플레이키 테스트는 미루지 말고
  즉시 원인(대부분 고정 sleep/공유 상태)을 잡는다 — 반복 실행으로 안정성 확인.
- **테스트는 밀폐한다**: 실키·실네트워크에 절대 의존하지 않는다 — 임시 config에
  `auto_providers: false`, 카탈로그 환경변수 키 격리. import 부작용(모듈 레벨 실행)이
  테스트를 오염시키지 않는지 확인한다.

## 문서-코드 정합

- **엔드포인트/CLI/설정 스키마를 바꾸는 커밋은 README·DESIGN.md·docs/IA.md 갱신을
  같은 브랜치에 포함한다.** 이 프로젝트 최다 결함원이 코드-문서 표류였다.
- 설계와 코드가 어긋나면 어느 쪽이 맞는지 먼저 결정하고, 문서부터 정합화한 뒤 코드를 만진다.

## 환경 특이사항 (Windows / PowerShell 5.1)

- 테스트 실행: `.venv\Scripts\python.exe -m unittest discover -s tests`
- git 인자(커밋 메시지 등)에 큰따옴표를 넣지 않는다 — PS 5.1 네이티브 인자 전달이 깨진다.
- 콘솔은 cp949 — **사용자 노출 문자열(로그 메시지, CLI 출력, Prometheus HELP)은
  영어 ASCII로**, 주석·독스트링은 한국어로 쓴다.
- 요청 경로 코드는 블로킹 I/O 금지(write-behind/to_thread), API 키는 로그·에러
  응답·대시보드 어디에도 노출 금지(마스킹).

# Required Artifacts (필수 산출물)

아래 문서들은 이 프로젝트의 필수 산출물이다. 각 문서는 존재해야 하며,
관련 작업을 진행할 때마다 최신 상태로 갱신한다. 누락된 문서를 발견하면
먼저 생성한다.

- `docs/ProjectContext.md` — 프로젝트 배경과 목표
- `docs/Plan.md` — 구현 전 작성하는 계획 문서
- `docs/IA.md` — 정보 구조(화면 목록, 화면 흐름)
- `docs/UserScenarios.md` — 사용자 시나리오
- `docs/Personas.md` — 사용자 페르소나
- `docs/DecisionLog.md` — 주요 의사결정과 근거
- `docs/WorkLog.md` — 작업 진행 내역, 오류와 수정 이력
- `docs/Process.md` — 작업 방식/프로세스(어떻게 일하는지에 대한 규칙과 절차)
- `docs/ReviewChecklist.md` — 완료 전 점검할 리뷰 체크리스트
- `docs/Research.md` — 기술/라이브러리 조사 및 리서치 내용
